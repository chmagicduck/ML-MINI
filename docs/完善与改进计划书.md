# 鱼额宝小程序（本地版）完善与改进计划书

更新时间：2026-03-11  
目标：在不引入后端的前提下，把现有“壳子”补齐为可持续使用的摸鱼工具，做到功能可达、数据可用、体验自洽，并通过引导页完成首次上手闭环。

## 1. 现状盘点（信息架构）

### 1.1 TabBar 与页面

当前 `miniprogram/app.json` 的页面与入口如下：

- `摸鱼中心`：`/pages/index/index`（计时 + 工资与进度）
- `情报中心`：`/pages/stats/index`（占位页，待开发）
- `实验室`：`/pages/lab/index`（会议烧钱机 / 拉粑粑 / 吃什么）
- `个人基地`：`/pages/profile/index`（等级路线图 + 简化统计 + 设置入口若干占位）
- `设置（基本设置）`：`/pages/settings/basic`（实际是“薪资与工时配置”）
- 其他子功能页：`/pages/poop/index`、`/pages/food/index`、`/pages/meeting/index`、`/pages/calendar/index`

### 1.2 已存在但不可达/未注册的页面

- 暂无（已清理）。

## 2. 功能闭环梳理（逐模块）

本章以“入口 -> 过程 -> 退出 -> 数据落盘 -> 展示/复盘 -> 再进入”作为闭环标准，列出已完成与缺口。

### 2.1 摸鱼中心（`/pages/index/index`）

已具备：

- 计时开始/暂停、跨天处理、增量提交到 `moyuStats`（按天聚合）
- 离线“工资入账”弹窗（基于 `lastExitState` 与已上班秒数差）
- 等级升级弹窗（含持久化 `pendingLevelUp`，避免漏弹）
- 工时进度、摸鱼率、发薪倒数/周末倒数、节假日提示（基于 `holiday.ts`）

闭环缺口与风险：

- 进度环 `progressStyle` 当前是基于 `slackingSeconds % 3600` 的视觉动效，并非真实进度表达；可以接受为动效，但命名与解释需要更明确，或改为“真实进度环 + 摸鱼层叠”。
- `_updateTimeUntilOffWork()` 仅在超时将 `offWorkPercent` 置 100，没有“倒计时/剩余时长”的闭环展示（注释与实际不一致）。
- 高刷新频率：摸鱼计时器 `100ms setInterval + setData`，长期运行可能造成掉帧/耗电；建议把 UI 刷新与数据落盘分离节流。

建议补齐项（闭环动作）：

- 开始/停止时增加“结算反馈”：本次新增秒数与新增收益（toast 或轻弹层），强化“我摸了多少”的回路。
- 增加“今日/本周”一键跳转到统计页（未来 `stats`），让数据有归宿。

### 2.2 情报中心（`/pages/stats/index`）

现状：

- 页面是占位文案，业务逻辑完全缺失。

闭环目标（建议实现为“个人战报中心”）：

- 汇总：今日/本周/本月摸鱼秒数、收益、摸鱼率趋势
- 可视化：近 30/90/365 天热力图（已具备 `moyuDaysMap` 数据结构基础）
- 对比：把目前 profile 的“击败同事/效率标签”从拍脑袋改成可解释算法（例如基于历史分位数的自洽指标）
- 行动：提供“去摸鱼/去实验室/去设置”的 CTA

### 2.3 实验室（`/pages/lab/index`）

现状：

- 入口完成（可进入会议/拉粑粑/吃什么）
- “已有 2,419 位组长在使用”等数据为硬编码展示（可保留为玩梗，但建议标注为“彩蛋”或改为本地统计）

建议闭环增强：

- 显示每个工具的“上次使用时间/累计收益/累计次数”摘要，形成“想起来就来”的闭环。

### 2.4 会议烧钱机（`/pages/meeting/index`）

已具备：

- 实时成本计算、参会人数控制
- 自定义人均月薪开关与输入
- 切后台保存状态并恢复（`meetingRunningState`）
- 支持分享文案

闭环缺口：

- 没有“会议历史记录”：用户无法回看“这周我被迫开了多少会，烧了多少钱”。
- 会议结束只停表，没有“结算摘要/保存一次会议记录”的动作。
- 对“未设置薪资”的提示存在，但计算用 `0`，体验上可接受；更好的闭环是引导去配置或提供默认值策略。

建议补齐项：

- 新增 `meetingStats`：累计会议时长、累计成本、会议记录列表（至少保留最近 N 条，类似 poop 的 `MAX_SESSIONS`）。
- 增加“结束并保存”与“仅暂停”区分，闭环成“开始 -> 结束 -> 记一笔 -> 报表”。

### 2.5 带薪拉粑粑（`/pages/poop/index`）

已具备：

- 计时开始/停止与保存一条 session（存到 `poopStats.sessions`）
- 切后台保存运行状态并恢复（`poopRunningState`）
- 会话条数上限（`MAX_SESSIONS = 200`）

闭环缺口与数据一致性问题：

- `PoopSession.date` 的类型约定是 `YYYY-MM-DD`，但当前实现为 `${year}-${month+1}-${day}`（不补零），会导致跨模块复用/排序/筛选时出现隐患。
- UI 标注“年度统计/历史总览”，但实际上只是“全量累计”，缺少按年/月的真实维度。
- 没有会话列表与删除/纠错机制（用户误触开始会产生脏数据）。

建议补齐项：

- 统一日期格式为 `YYYY-MM-DD`（补零），并考虑对旧数据做兼容迁移（见第 3 章数据治理）。
- 增加“最近记录”列表（最近 10 次），以及“删除最近一次”的安全兜底。

### 2.6 今天吃什么（`/pages/food/index`）

已具备：

- 抽奖动效、最近 5 次历史（仅内存，页面生命周期内）
- 分享

闭环缺口：

- “✅ 就这个了！”按钮无交互，缺少“确认/锁定/记录”的闭环动作。
- 历史不落盘，退出页面即丢失，无法形成长期偏好或统计。

建议补齐项：

- 确认按钮：落盘到 `foodHistory`，并 toast “已记录，祝你吃好”；可扩展“避开最近 N 次”策略。
- 引入“自定义菜单”：本地增删条目、权重、黑名单。

### 2.7 节假日日历（`/pages/calendar/index` + `utils/holiday.ts`）

已具备：

- 月视图渲染、翻月、图例、发薪倒数

闭环风险：

- `holiday.ts` 自述为“推算，精确以公告为准”，且仅覆盖 2026；当年份变化时（例如 2027）会退化为“只按周末判断”的粗粒度逻辑，影响：首页提示、工作日计算、秒薪计算、工时进度等多个模块。

建议补齐项：

- 明确策略二选一：
  - A. 彻底离线：内置 2024-2028（或更多）官方日历数据文件，按年切换；提供“数据版本号”和“提示免责声明”。
  - B. 半离线：允许用户手动导入/更新节假日数据（json），默认提供 2026。
- 日历支持“点某天看详情”（假/班/周末/发薪日），并可一键复制“还有几天发薪”文案。

### 2.8 个人基地（`/pages/profile/index`）

已具备：

- 累计收益展示与等级进度
- 路线图展示
- 数据重置（清 storage 多个 key）
- 跳转到 `基本设置`、`日历`

闭环缺口：

- “外观皮肤 / 时钟样式 / 关于软件”均为不可点击占位项，没有对应页面与逻辑。
- 皮肤系统在 `types.ts/storage.ts` 已定义，但 assets 中没有 `assets/images/skins/*`，也没有任何地方读取并应用皮肤（`getCurrentSkin()` 未被使用）。
- “击败同事/摸鱼效率”指标为简化算法，并非基于可解释数据（尤其“击败同事”目前取决于 `moyuDays`，容易失真）。
- 头像与昵称固定写死为“微信用户”，没有个性化闭环（可以不接入用户信息，但至少要有“自定义昵称”本地设置）。

建议补齐项：

- 先做“可落地”的闭环：把“外观皮肤/关于软件”做成能进入的页面，哪怕是简版。
- 把 profile 的指标迁移到 stats 页，profile 保持“摘要 + 快捷入口”，避免堆叠假数据。

## 3. 数据与配置治理（让功能可维护）

### 3.1 当前 storage key（建议统一文档化）

- `userSettings`：用户配置（薪资/时间/工作制/音效等）
- `moyuStats`：累计摸鱼统计（总收益/总秒数/按天 map）
- `slackingToday_YYYY-MM-DD`：当天摸鱼秒数（用于首页计时恢复）
- `pendingLevelUp`：升级弹窗待展示
- `lastExitState`：离线工资弹窗状态
- `initialIdentityShown`：首次身份弹窗是否展示
- `poopStats`：拉粑粑统计（总秒数/总收益/sessions）
- `poopRunningState`：拉粑粑计时恢复
- `meetingRunningState`：会议计时恢复
- `currentSkin`：当前皮肤 id（未应用）

### 3.2 版本化与迁移（闭环必备）

建议增加：

- `storageVersion`：数据结构版本号
- 启动时迁移：例如修复 `poopStats.sessions[].date` 的补零格式，或后续新增 `meetingStats` 的默认值补齐

> 不做迁移的风险：一旦上线新功能或修 bug，老用户数据会出现不可预期分支，导致“统计不准/崩溃/显示空白”。

当前设置统一收敛为：

- `薪资与工时`：`/pages/settings/basic`

建议后续若要扩展更多开关（例如外观/实验性功能），再新增独立设置页或将其并入个人基地的设置分组，避免出现多套入口并存。

## 5. 引导页（Onboarding）规划

### 5.1 引导页的目标

- 首次进入 30 秒内完成“关键配置闭环”：
  - 配置月薪、上下班时间、午休、工作制、发薪日
  - 明确“摸鱼计时只在工作时段可开启”的规则
- 让用户知道主功能在哪里：摸鱼中心、实验室、个人基地/战报
- 对可选项给默认：隐私提示（本地存储、不上传）

### 5.2 触发策略（推荐）

- 首次启动：若 `hasOnboarded !== true`，且启动目标页面是首页（而非分享直达页），则 `reLaunch` 到引导页。
- 未配置关键项：若 `monthlySalary <= 0`，也触发“轻量引导”（可直接跳到“快速设置”步骤）。
- 可重复访问：在 profile/settings 中提供“重新引导/新手教程”入口。

### 5.3 信息架构与页面形态

推荐新增页面：`/pages/guide/index`

- Step 1：欢迎 + “本地应用，不上传数据”说明
- Step 2：三句话说明四个 tab 的用途（摸鱼中心/情报/实验室/个人基地）
- Step 3：快速设置（内嵌关键表单，或复用 basic 的表单结构）
- Step 4：完成页：给一个“开始摸鱼”按钮（跳首页），并提示“设置可随时修改”

### 5.4 技术落地清单（最小可行）

- 新增 storage：
  - `hasOnboarded: boolean`
  - `onboardingVersion: string`（例如 `2.6.0`，大改时可再次触发）
- `App.onLaunch` 做路由守卫：
  - 读取 launch options，判断是否需要跳转引导页
  - 避免从分享卡片进入子功能时被强制拦截（可在子页顶部用 banner 引导“去完成配置”）
- 引导页完成时：
  - 保存 settings（至少保证 `monthlySalary` 与工时完整）
  - 置 `hasOnboarded=true`，`reLaunch('/pages/index/index')`

## 7. 功能检查与验收清单（手工回归用）

用于“检查所有功能”的闭环验证，建议每次发版前至少跑一遍 P0 项。

### 7.1 基础设置（`/pages/settings/basic`）

- 能正常保存月薪、发薪日、上下班、午休、工作制；返回后首页立即刷新秒薪/本月工作日数/发薪倒数
- 校验生效：
  - 月薪为空或 <= 0 提示
  - 上班时间 >= 下班时间提示
  - 午休启用时，午休开始 >= 结束提示

### 7.2 摸鱼中心（`/pages/index/index`）

- 首次启动：
  - 未配置月薪时，提示条出现；基础数值不崩
  - 第一次开始摸鱼，会弹“初始身份”且只弹一次
- 正常计时：
  - 开始后计时增长；暂停后停止增长；再次开始可续上
  - 关闭小程序再打开：若此前处于计时中，能恢复“计时仍开启”的状态（或按设计改为自动暂停，但需一致）
- 数据落盘：
  - 暂停后 `moyuStats.totalSeconds/totalMoney` 有增长
  - 当天 `slackingToday_YYYY-MM-DD` 存在并与 UI 一致
- 跨天：
  - 23:59 开始，跨到次日：前一天数据应被归档，次日从 0 开始（且无异常弹窗连发）
- 工作时间限制：
  - 工作时段外点开始，提示“无法开启摸鱼”
  - 工作时段结束自动停表（toast 提示）
- 离线工资弹窗：
  - 退出 >= 1 小时、同一天再进入，会弹一次“你不在的时候...”并不会重复弹

### 7.3 会议烧钱机（`/pages/meeting/index`）

- 自定义人均月薪开关：
  - 开启输入有效时，成本按自定义计算

- 切后台恢复：
  - 运行中切后台 10 秒再回来，时间与成本应补齐，不发生跳变（人数/自定义薪资上下文保持）
- 分享：
  - `open-type="share"` 可触发分享文案回调

### 7.4 带薪拉粑粑（`/pages/poop/index`）

- sessions 上限：反复记录直至超过 200，仍能正常使用且不会崩（旧记录被截断）

## 8. 需要你确认的产品决策（不确认也能做，但会影响实现路径）

1. 皮肤系统你希望做成“可用功能”（需要补齐 skins 资源与应用逻辑），还是暂时移除所有相关 UI 占位，等后续再做？
